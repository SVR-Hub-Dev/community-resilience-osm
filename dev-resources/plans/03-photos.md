# Plan: 03 — Photos (R2 binding, server-side helpers, dev secrets)

## Context

[02-data-layer.md](02-data-layer.md) landed the `photo` table (`storage_path`, `taken_at`, `caption`, FKs to `asset` and `inspection`) but nothing in the repo can yet write a byte to R2 or hand the browser a URL for an existing object. The Kobo webhook (architecture §6.2), the asset detail panel (§5.1), and any future inspection upload UI all sit on top of a missing layer: an R2 binding declared in [wrangler.jsonc](../../wrangler.jsonc) and a small server-only module that wraps `put` / `get` / presigned-read against `platform.env.PHOTOS_BUCKET`. This plan adds exactly that — binding, helper, dev secrets — and stops there. The webhook handler that consumes it is its own plan (Kobo ingestion), and the map UI that displays photos is `04-map-ui.md`.

## Scope

- [wrangler.jsonc](../../wrangler.jsonc) — declare the R2 binding (`PHOTOS_BUCKET` → bucket `photos`) per architecture §8.2.
- [worker-configuration.d.ts](../../worker-configuration.d.ts) — regenerated by `npm run gen` so `platform.env.PHOTOS_BUCKET` types as `R2Bucket`.
- [src/lib/server/r2.ts](../../src/lib/server/r2.ts) — server-only module exporting:
  - `putPhoto(bucket, key, body, contentType)` — thin wrapper over `bucket.put` that pins `httpMetadata.contentType` and returns the object key. The webhook plan calls this; centralising it keeps key-construction conventions in one place.
  - `getPresignedPhotoUrl(env, key, ttlSeconds)` — generates a short-lived signed GET URL using the S3-compatible API (architecture §8.3). The binding itself does not sign URLs; signing happens via `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` against `https://<R2_ACCOUNT_ID>.r2.cloudflarestorage.com`.
  - `photoKey({ inspectionId, slot })` — pure helper that produces `inspections/<uuid>/<slot>.jpg`, matching the architecture §6.2 example. Pure function, no I/O, easy to unit-test.
- New runtime deps: `@aws-sdk/client-s3`, `@aws-sdk/s3-request-presigner`. Both run inside Cloudflare Workers (HTTP-only signing, no Node `net` / `tls`).
- [.dev.vars.example](../../.dev.vars.example) — append `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`. The binding itself needs no secrets, but `getPresignedPhotoUrl` does (the SDK signs against the S3 endpoint, not the binding).
- [README.md](../../README.md) — extend the existing setup section with three lines: "Create the R2 bucket named `photos` in the Cloudflare dashboard, generate an R2 API token, paste the three R2 values into `.dev.vars`."
- One unit test, [src/lib/server/r2.test.ts](../../src/lib/server/r2.test.ts), covering `photoKey()` only — pure function, no Worker / Miniflare setup needed. The signed-URL path is exercised via Playwright in a later plan.

## Out of scope

- The Kobo webhook handler (`/api/kobo-webhook/+server.ts`) — its own plan; it will import `putPhoto` and `photoKey` from the module landed here.
- Any `+server.ts` route that returns photo URLs to the browser, and the asset detail panel that consumes them — `04-map-ui.md`.
- Image transforms / thumbnailing / Cloudflare Images — out of architecture scope for MVP. If we want resized variants later, that's an ADR.
- Public-bucket mode (architecture §8.3 alternative) — we go with presigned URLs only for the MVP. Adopting public mode later is a config flip + a `R2_PUBLIC_BASE_URL` env var.
- Bucket lifecycle rules / retention / cold-tier moves — runbook, not code.
- Per-row authorisation (who can see which photo) — gated on the auth provider ADR (architecture §3.2). The presigned URL is short-lived; visibility is decided in the load function before signing.
- Photo upload UI from the browser (direct-to-R2 with a presigned PUT) — possible later optimisation; the MVP path is Kobo → webhook → R2.

## Approach

**R2 binding vs. S3 SDK — both, with discipline.** Architecture §8.2 calls binding-style "preferred when running inside the Worker" and reserves the S3 SDK for code that runs outside (or when signing presigned URLs). We follow that split exactly: writes go via `bucket.put` (the binding); read URLs go via the SDK (signing). `getPresignedPhotoUrl` takes `env` rather than a `bucket` argument because it does not touch the binding at all — it talks to the R2 S3 endpoint. Keeping the two functions in the same file is fine; they share the conceptual unit "how this app touches R2."

**Why not implement SigV4 by hand.** The AWS SDK adds ~100 KB to the Worker bundle, but it is the documented presigning path, it survives R2 endpoint changes, and the alternative (a hand-rolled HMAC-SHA256 SigV4 signer) is the kind of crypto code that is easy to subtly miswrite and hard to detect when it goes wrong (URLs work in dev, fail in prod after some clock skew). Pay the ~100 KB once; revisit only if bundle-size becomes a real constraint.

**`platform` typing.** SvelteKit's adapter-cloudflare populates `App.Platform` from `worker-configuration.d.ts`. The existing [src/app.d.ts](../../src/app.d.ts) already references `Cloudflare.Env`; regenerating types after the binding change is the only step needed for `platform.env.PHOTOS_BUCKET` to type as `R2Bucket` everywhere.

**Key construction.** Architecture §6.2 hard-codes `inspections/${insp.id}/${key}.jpg`. Pulling that into `photoKey()` removes a small footgun: future callers that want a different prefix (e.g. `assets/<ref>/cover.jpg` for a non-inspection cover photo) can extend the helper rather than scatter string templates. The MVP only needs the inspection variant; `photoKey` exposes a discriminated union so adding `{ assetRef, slot: 'cover' }` later is type-safe.

**Presigned URL TTL.** Default to 5 minutes (`300`). Long enough for a slow page render and an `<img>` fetch, short enough that a leaked URL is harmless within minutes. Pass `ttlSeconds` explicitly so `+page.server.ts` callers can shorten it for sensitive-marked assets without touching this module.

**Local dev.** `wrangler dev` ships a local R2 simulator that the binding routes to automatically — `bucket.put` writes to `.wrangler/state/v3/r2/`. The S3 SDK path (presigned URLs) talks to the _real_ R2 endpoint; in local dev that means real R2 credentials are required for read URLs to work end-to-end. Document this in the README so a developer doesn't burn an hour wondering why writes succeed but reads 403.

**No `r2.test.ts` browser harness.** Testing `putPhoto` and `getPresignedPhotoUrl` properly requires Miniflare (for the binding) plus a fake R2 endpoint (for the SDK). That's a lot of harness for two thin wrappers. Defer integration coverage to the Playwright suite once the webhook plan and detail-page plan land — by then the path is exercised end-to-end naturally. `photoKey` is the only piece worth a unit test in isolation.

- No browser-facing changes — nothing to verify in Playwright this phase.

## Follow-ups

- **Kobo webhook plan** — `/api/kobo-webhook/+server.ts` consuming `putPhoto` / `photoKey`. Unblocked by this plan.
- **`04-map-ui.md`** — the asset detail panel will call `getPresignedPhotoUrl` from the page's `+page.server.ts` load function. Unblocked.
- **ADR candidate — public bucket vs. presigned URLs**: this plan commits to presigned-only. If we later want a public photo gallery for non-sensitive assets, document the cutover (bucket public flag + a `R2_PUBLIC_BASE_URL` env var + branch in `getPresignedPhotoUrl`).
- **Runbook**: `dev-resources/runbooks/r2-bucket-setup.md` — bucket creation, API token issuance, what scope the token needs (Object Read & Write), how to rotate it. Currently the runbooks directory is still empty.
- **Bundle-size budget**: with two AWS SDK packages in the Worker bundle, this is a good moment to start tracking compiled-Worker size. Not a blocker; capture as an issue.
- **Image transforms** (resize / WebP) — if photos balloon past the R2 free tier, look at Cloudflare Images or an on-the-fly transform. Out of MVP scope.
- **Direct-to-R2 browser uploads** — possible if a non-Kobo upload path lands later. The architecture path stays: presigned PUT URLs signed by the same SDK, scoped to a single key.
